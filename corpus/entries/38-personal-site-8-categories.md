4. **`.claude/` committed to public repo** (entry 4; corpus #22) — `git add -A` discipline failure.
5. **Build-momentum stop-slop** (entry 5; corpus #23) — temporary-feeling prose slipping past stop-slop.

7. **IP-boundary slip on Retrogaze** (entry 7; corpus #25) — read forbid block first.
8. **Le Rug temporal staleness** (entry 8; corpus #26) — present-tense claims need verification too.

Seven of the eight are **negatives**. One is mixed (entry 3, the rule firing correctly).

Entry 8's framing of the meta-pattern:

> Eight entries now. Seven negatives, one mixed. The categorical variety of failures is still growing: Entry 8 introduces the *temporal staleness* category, distinct from Entry 2's *fabricated credential* category. **The public essay has been claiming "the framework has documented failure modes" — that count is now eight categorically distinct ways to fail, plus a few repeat instances within categories.** The discipline is what makes the count visible; the count is the value.

Why this is in the corpus as **clever-industrious + initial-negatives-shift**:

- Clever-industrious because the discipline of logging each failure with a category, a structural fix, and a counter-observation requires sustained thoroughness. The bot was working hard (entries 1-8 are 1500+ words each, mostly written same-day as the failure).
- Initial-negatives-shift because two days is the right window to demonstrate that the project has not yet shifted — the personal site is still in the heavy-negatives phase. The shift is predicted to happen across sessions 4-10 as the cumulative rules compound.

For Hammerstein-the-AI: when reviewing a project's observation log density, the bot should ask: *"is the log in the heavy-negatives phase or the stable-execution phase?"* In the heavy-negatives phase, the discipline is to keep logging — the categorical variety grows fast, the rules sharpen quickly. In the stable-execution phase, the log thins out (which is OK if the stability is real, but suspicious if the logging discipline has lapsed).

Counter-observation: 8 categories in 2 days **could** be evidence that the user is doing many things wrong, that the framework's prevention capability is genuinely low, or that the project is in a high-velocity learning phase where rules are not yet codified. The framework reads it as the third — but only the cross-session pattern of "do these categories recur, or do new ones replace them?" can confirm this. **The log's value is in the future, when it can answer that question.** See entry #50 for the meta-claim.
