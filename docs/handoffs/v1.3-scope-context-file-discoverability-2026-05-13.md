## Why this exists

A 2026-05-13 Hammerstein audit (`hammerstein "<HAi Phase 2A scope>" --template audit-this-plan`) recommended adding Cloudflare KV as a rate-limit state layer because the audit had **no access to the actual `wargamer.ts` code** that was being modified. The wargamer.ts already used Cloudflare edge cache for the same purpose. The recommendation was wrong-shaped — over-built — but the underlying complaint ("the plan didn't make storage explicit") was right.

Filed as `hai-036` in `state/hammerstein-ai/tasks.json` (GS-private), cross-referenced here.

## What v1.2 already shipped (don't rebuild)

Per `CHANGELOG.md` 1.2 (2026-05-12):

- `hammerstein --context none|minimal` — opt-in project-context injection (bounded preamble with repo identity + capped excerpts from MISSION.md / CLAUDE.md / AGENTS.md / README.md)
- `--context-file <path>` — explicit context preamble file (preferred grounding mechanism)
- `--project-root <path>` — override git-root detection
- Security: denylists sensitive filenames + scrubs absolute paths + aborts on high-entropy credential patterns

So the **flag exists.** The 2026-05-13 incident was the operator (parent CC) not USING it — not a feature gap.

## What v1.3 should address

Two complementary surfaces, ordered by ROI:

### Surface A — Discoverability: make the flag harder to forget [HIGH ROI / LOW COST]

The operational gap is real and reproducible: I (parent CC, drive-mode HAi sessions) ran `hammerstein "<long plan referencing wargamer.ts>" --template audit-this-plan` without `--context-file`, and got an audit that recommended over-built infrastructure because it couldn't see the code. The fix is to make the operator notice when context would help.

**Option A1 — Detect file mentions in the plan text + warn.**

If the plan text contains path-shaped tokens (`functions/api/wargamer.ts`, `src/foo.rs`, etc.) AND `--context-file` isn't passed AND `--context` isn't explicitly `none`, print a one-line stderr warning before dispatching:

```
hammerstein: plan references 3 file paths; consider --context-file or --context minimal for grounded audit
  detected: functions/api/wargamer.ts, scripts/mint-token.ts, site/wargamer/index.html
```

Non-blocking; the audit still fires. Just a nudge so the operator doesn't ship an unrooted audit by accident. Behind the existing `--context none` opt-out.

**Implementation surface:** ~20 lines in `hammerstein_cli/__init__.py` or wherever query dispatch lives. Regex-detect path-shaped tokens (basic heuristic — `[a-z_]+\/[a-z_]+\.(ts|js|py|rs|go|md)` etc.); cross-reference against `Path.exists()` from project root if known; print warning if N≥1.

**Risk:** False positives (an audit prose that just mentions "src/foo.ts" generically). Mitigation: gate on `Path.exists()` so we only warn when the file actually exists in the project.

### Surface B — Auto-attach: pull referenced files into context automatically [MEDIUM ROI / MEDIUM COST]

Once Surface A is shipped + validated, the next step: if the operator opts in (e.g. `--auto-context` flag or template-level default for audit-this-plan), automatically pull the referenced files into the context preamble, subject to:

- File-exists check against project root
- Total budget cap (e.g. 4-8k chars across all auto-attached files)
- Per-file cap (e.g. 1500 chars head + 1500 chars tail; or signatures-mode-style compression)
- Same security gates as `--context minimal` (denylist sensitive filenames; abort on credential patterns)

**Implementation surface:** ~80-120 lines extending the existing context-injection module. Reuse the security gates already in place.

**Risk:** Larger audit prompts → more OpenRouter cost per audit. Mitigation: cap the budget; surface the auto-attached files in `--show-prompt` so operator can sanity-check before paying for real audits.

### Surface C — MCP file-read tool [HIGH ROI / HIGH COST]

A genuine MCP server exposing a `read_file(path: string) -> string` tool that Hammerstein's model can call during audit reasoning. This lets the model itself decide which files to read, instead of the operator pre-attaching them.

Out of scope for v1.3 — non-trivial implementation, opens new failure modes (model decides to read 47 files, blows the budget). Defer to v1.4+ if Surface A + B don't close the gap.

## Recommended sequence

1. **Ship Surface A** as v1.3.0 — warning-only, ~30 min implementation + tests
2. **Validate operationally** for 1-2 weeks: does the warning fire on real audits I run? Does it reduce ungrounded-audit incidents?
3. **Ship Surface B** as v1.3.1 — opt-in auto-attach, if Surface A's warning proves the gap is real and persistent
4. **Defer Surface C** to v1.4+ — MCP tool integration is a bigger architectural call

## Open questions for the picking-up CC session

1. **Warning verbosity.** Lock at one stderr line, or more elaborate (list paths, suggest exact command, etc.)? Lean: one line, low-noise.
2. **Heuristic for "path-shaped token."** Lock the regex: `([a-zA-Z0-9_./-]+\.(ts|tsx|js|jsx|py|rs|go|md|sql|yaml|yml|toml|html|css))` is a reasonable starting set. Override if too narrow / broad.
3. **Surface B trigger.** Auto-attach behind a new flag (`--auto-context`) vs auto-on-for-audit-this-plan-only vs always-on-with-opt-out? Lean: new flag, opt-in. Don't change existing audit cost-profile without operator intent.
4. **Budget cap.** Surface B's total char budget. Lean: 6k chars total across all auto-attached files (matches v1.2's 2.2k project-context cap doubled-plus, since file contents are higher-signal than MISSION.md excerpts).

## Hands-off awareness (when picking this up in a hammerstein-CLI session)

- **`harness/hammerstein/cli.py`** is the argparse entry; the typer entry at `hammerstein_cli/__init__.py` is the installed `hammerstein` command. Changes need to flow through BOTH unless they're typer-only operational UX (warnings are typer-only fine).
- **`prompts/templates/audit-this-plan.md`** is the few-shot template. Surface A + B don't touch it.
- **Security gates** in context injection are load-bearing — the denylist + path-scrub + credential-abort logic in v1.2's context module should NOT be relaxed to make auto-attach more permissive. Reuse the existing gates as-is.
- **No changes** to `corpus/`, `prompts/system.md`, or `eval/` — this is a CLI-ergonomics surface, not a framework-definition or methodology surface.
- **Tests** in `tests/` should cover: warning fires on path-mention without `--context-file`; warning doesn't fire with `--context none`; warning doesn't fire when paths don't exist; auto-attach respects budget cap; auto-attach respects denylist; auto-attach skips on credential patterns.
